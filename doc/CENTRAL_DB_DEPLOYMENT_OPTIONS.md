## 0. The locked profile (read this first)

This document answers a **specific** profile. Every recommendation below is conditioned on these four locked decisions — they are not up for re-litigation here.

| Axis | Locked decision | What it rules OUT | What it rules IN |
|---|---|---|---|
| **Hosting** | **Self-hosted Postgres on the user's OWN infra/cloud.** "Self-hosted" = Docker-Compose Postgres on a VM, *or* a Postgres the user runs in their own cloud account (AWS RDS / GCP Cloud SQL / Fly Postgres / a k8s Postgres operator). The point is the **user controls it and no third party holds the data**. | No managed memory SaaS. No managed Supabase. No Supavisor. No third-party-held data. | **WE** install pgvector. **WE** run pgbouncer/pooling. **WE** own backups/PITR. **WE** set `max_connections`. |
| **Trust** | **Strict, human-gated.** Only human-sourced content (ticket Q&A answers, human-EM PR-approval decisions, board-promoted entries) is retrievable as AUTHORITATIVE. Agent-generated claims are written but forced `unverified` and quarantined until a human promotes/verifies. Default, not optional — but rolled out **label-then-exclude** to avoid retrieval starvation. | Agent self-assertion of "verified". Retrieval-only filtering as the sole control. | A write-side gate (`createEntry`, `server/src/services/memory.ts:228`) **plus** a retrieval-side filter. |
| **Tenancy** | **Multi-team org** — several companies under one self-hosted instance, each isolated. Therefore **Postgres RLS is REQUIRED** before team/multi-company onboarding; it is not a someday-maybe. App-layer `companyId` scoping is the bridge for the single-operator phase only. | Shipping multi-tenant on app-layer scoping alone. | RLS migration (`0053`), `SET LOCAL`-inside-txn, a `BYPASSRLS` background role. |
| **Knowledge UI** | **Postgres + the existing ADE memory UI is the SINGLE system of record.** Obsidian/markdown is NOT in the plan (read-only export is at most a deferred nicety; bidirectional sync is rejected as a staleness/hallucination vector). | Re-platforming onto Obsidian/Notion/Pinecone/Neo4j/mem0. Bidirectional markdown sync. | `ui/src/pages/CompanyMemory.tsx` + `memory_entries` as SoR. |

**Timeline shape:** *individual-today → team-later.* One operator dogfoods today (embedded Postgres, zero deploy); a self-hosted central DB is the first real deploy; multi-team RLS lands at the team-onboarding boundary as a hard gate.

**Code facts that constrain everything below** (verified against the repo at HEAD migration `0047`):

- **Boot auto-applies migrations with no advisory lock.** External boot calls `ensureMigrations(config.databaseUrl, "PostgreSQL")` with no `autoApply` (`server/src/index.ts:278`), which falls through to `promptApplyMigrations`, and that returns `true` whenever `!stdin.isTTY || !stdout.isTTY` (`server/src/index.ts:135`) — so **a headless container silently auto-applies on every boot.** `applyPendingMigrations` runs `migratePg` on a dedicated `max:1` connection but takes **no `pg_advisory_lock`** (`packages/db/src/client.ts:647-681`).
- **Pooler detection is Supabase-specific.** `pgOptions` sets `prepare:false` **only** when the URL port is exactly `"6543"` (`packages/db/src/client.ts:14-17`). A self-hosted pgbouncer on any other port/host gets `prepare:true` and breaks under transaction pooling.
- **Pool sizing is implicit.** `createDb` calls `postgres(url, pgOptions(url))` with **no `max`** (`packages/db/src/client.ts:50-53`) → silent postgres-js default `max:10` per process. Maintenance paths use `max:1`.
- **Zero RLS today.** `grep -rn "POLICY\|ENABLE ROW LEVEL\|BYPASSRLS" packages/db/src/migrations/` returns nothing. The only fence is app-layer `assertCompanyAccess` (`server/src/routes/authz.ts:18`), which **short-circuits for `local_implicit` and `isInstanceAdmin`** (`authz.ts:12`, `:25`).
- **A global cross-company scan exists.** `heartbeat.ts:6559` runs `const allAgents = await db.select().from(agents);` with no company filter — this returns **zero rows** under naive RLS unless run as a `BYPASSRLS` role.
- **Advisory-lock primitive is the non-blocking variant on a pooled connection.** `summarizer-failures.ts:165` uses `pg_try_advisory_lock`, and `summarizer-queue.ts:15-19` documents the hazard verbatim: *"`pg_try_advisory_lock` is session-scoped and postgres-js pools connections, so lock/release can land on different sessions. Proper integration needs a reserved client per call."*
- **The repo already ships a near-complete compose topology.** `docker-compose.yml` defines a `db` service (`image: postgres:17-alpine`, healthcheck, `pgdata` volume) and a `server` service pointed at `postgres://combyne:combyne@db:5432/combyne`, with `DATABASE_URL` and `BETTER_AUTH_SECRET` inlined as **plaintext env**. There is one `Dockerfile` and `docker-compose.quickstart.yml`; **zero k8s manifests / Helm charts.**
- **The cutover has no memory ETL yet.** `company-portability.ts` carries **no** memory table (config-only bundle, synthesis §8.4), so switching `DATABASE_URL` today silently boots an **empty** central DB and loses all dogfooded memory. The `db:memory-export`/`db:memory-import` scripts (synthesis §8.4) are a **hard prerequisite** for any topology move.
- **`DATABASE_URL` resolution.** `databaseUrl: process.env.DATABASE_URL ?? fileDbUrl` (`server/src/config.ts:241`); unset → embedded Postgres auto-starts at `127.0.0.1:54329` (`index.ts:284`). Backup cadence is config-driven: `databaseBackupEnabled` / `databaseBackupIntervalMinutes` / `databaseBackupRetentionDays` (`config.ts:49-51`, `:191-201`).

## 1. Deployment topology — option families

### 1.1 Topology comparison matrix

| | **A. Docker-Compose Postgres on a single VM** | **B. Managed Postgres in the user's OWN cloud account (RDS / Cloud SQL / Fly PG)** | **C. Kubernetes Postgres operator (CloudNativePG / Zalando)** | **D. Primary + read-replica** (overlay on B or C) |
|---|---|---|---|---|
| **What it is** | One VM runs `docker compose`: a Postgres container + the Combyne `server` container (the repo's existing `docker-compose.yml`, image swapped to `pgvector/pgvector:pg17`). | Postgres provisioned as a managed service *inside the user's own account* — still self-hosted in the locked sense (no third party holds data; user owns the account + kill switch). | Postgres inside the user's own k8s cluster via an operator: primary + replicas, leader-election failover, declarative cluster spec. | Not a standalone choice — a **scaling overlay**. Add a streaming read replica; route the hot read path (retrieval) to it, writes/migrations to the primary. |
| **Ops / migrations** | App auto-migrates on boot (`index.ts:135`); at single replica this is **safe** (nothing races). | Run a one-shot `pnpm db:migrate` against the direct endpoint; do NOT let >1 app replica auto-apply. | **Must** gate boot migrations (`COMBYNE_RUN_MIGRATIONS_ON_BOOT=false`) + one-shot Job + a blocking `pg_advisory_lock` the moment the Deployment scales past `replicas:1`. | Migrations + writes go to the primary only; a careless split sends a write to the replica and fails. |
| **Pooling** | None needed at one operator: `max:10` app pool + ~6 transient `max:1` maintenance paths, trivially under a self-set `max_connections=100`. The hardcoded `port===6543` prepared-statement disable (`client.ts:14-17`) never fires (no pgbouncer in path). | WE run pgbouncer (transaction mode) or the provider proxy. Requires the robust-pooler-detection fix first (replace `port===6543`). | CloudNativePG ships a managed pgbouncer (`Pooler` CRD) — the self-hosted Supavisor replacement under our control. | Every replica multiplies `app_pool_max × replicas` against our chosen `max_connections`. |
| **Backups / PITR** | WE own it end to end: pinned `pg_dump` cron (the existing `db:backup` path) + WAL archiving to object storage for PITR. Failover = restore (no HA). | Automated snapshots + PITR are checkboxes **inside our account** (RDS/Cloud SQL). Fly Postgres is the exception — it does NOT give RDS-grade automated backups. | Continuous backup to object storage (Barman/CloudNativePG) → real PITR without hand-scripting. | Inherits the primary's backup story. |
| **Tenant isolation** | RLS applies identically; one VM, one operator → app-layer scoping is sufficient until team. | RLS applies; validate `SET LOCAL`-in-txn against whichever proxy you choose. | First-class RLS substrate for true multi-company; per-tenant isolation is a hard requirement at this scale. | RLS policies replicate; the `BYPASSRLS` role + `SET LOCAL` scoping must be validated on the read path too. |
| **Vector / embeddings** | pgvector is a one-line image swap (`pgvector/pgvector:pg17`); migration `0052` lands with zero new infra. | `CREATE EXTENSION vector` (RDS PG15+, Cloud SQL, Fly pgvector image). | Operator image bundles pgvector or via init/extension config. | ANN (HNSW) scans are read-only + CPU-heavy → the exact workload to push to a replica once `0052` lands. |

### 1.2 Per-option PROS / CONS

#### A. Docker-Compose Postgres on a single VM

| PROS | CONS |
|---|---|
| Lowest cognitive + operational burden — literally the repo's existing `docker-compose.yml` with the image swapped to `pgvector/pgvector:pg17`. User controls 100% of the data on their own VM (satisfies self-hosted exactly). | **No HA.** VM/disk loss = downtime until restore. RTO = restore time; RPO = WAL-archive cadence. Acceptable for one operator, marginal for a daily-dependent team. |
| Single-replica makes the boot-time auto-migration (`index.ts:135`) **actually safe** — no second replica to race, so the missing advisory lock (`client.ts:647-681`) is not yet a hazard. | You own backups/PITR end to end. `company-portability.ts` carries no memory tables (synthesis §8.4), so a DB-level backup is **mandatory, not optional** — and nothing in the repo configures WAL archiving for you. |
| No pooler needed at one-operator scale; the hardcoded `port===6543` quirk (`client.ts:14-17`) is irrelevant with no pgbouncer in the path → prepared statements work normally. | Co-locating app + DB on one box couples their failure + resource domains; a runaway agent run can starve Postgres of CPU/IO. |
| pgvector install is one line; the deferred pgvector migration (`0052`) lands here with zero new infra. | Vertical scaling only — growing read load means resizing the VM (downtime), not adding a replica. |
| Cheapest possible: one small VM ($10-40/mo) hosts both app + DB; no managed markup, no per-connection billing. | Self-managed Postgres tuning (`shared_buffers`, `work_mem`, `max_connections`, autovacuum) is now your job; the alpine default config is not production-tuned. |

**Best when:** the one-operator-today phase and the immediate central-DB cutover. **Start here** — the smallest step beyond embedded that yields a durable, network-reachable, pgvector-capable Postgres the user fully controls.

#### B. Managed Postgres in the user's own cloud account (RDS / Cloud SQL / Fly Postgres)

| PROS | CONS |
|---|---|
| Backups/PITR + OS patching become the provider's problem **inside YOUR account** — automated snapshots + PITR are checkboxes, removing option A's single biggest liability (untested backups). | The repo's pooler detection is hardcoded to `port===6543` (`client.ts:14-17`). Fronting the DB with pgbouncer (or RDS Proxy) in **transaction mode** breaks prepared statements unless you ship the robust detection first — a **hard prerequisite**. |
| Built-in failover (RDS Multi-AZ / Cloud SQL HA) promotes a standby with no app change — real durability for the team phase without you running an HA operator. | Cost is higher than a single VM: a small instance ~$15-50/mo; Multi-AZ roughly doubles it; storage + IO + egress on top. Still cheap, not free. |
| pgvector is first-class: `CREATE EXTENSION vector` per database. Migration `0052` is one statement away. | **Fly Postgres is operationally closer to A** — it does NOT ship RDS-grade automated backups/PITR/failover. Choosing Fly silently inherits A's liabilities while you believe you bought B's safety. This must be stated explicitly. |
| You control `max_connections` directly via the instance parameter group and run the provider's proxy OR your own pgbouncer — your call, your account, no external Supavisor. | Provider connection-proxy semantics differ from pgbouncer; the `SET LOCAL`-inside-txn requirement for RLS (synthesis §9.2; pinning hazard `summarizer-queue.ts:15-19`) must be validated against whichever proxy you choose. |
| Clean separation of app VM and DB removes A's resource-contention failure mode; the app can be a tiny instance. | You still must NOT let multiple app replicas auto-apply on boot (`index.ts:135` race). Run one-shot `pnpm db:migrate` against the direct endpoint. |
| Smoothest one-operator→team growth: bump the instance class + add a read replica without re-architecting. | Assumes the user is on AWS/GCP. On a provider without an RDS/Cloud SQL equivalent (or ideologically on-prem), B collapses toward A-with-extra-cost. |

**Best when:** the central-DB cutover when durability/PITR/failover matter and the user is on AWS/GCP — the **recommended landing spot for the team phase.** Pick RDS or Cloud SQL (not Fly Postgres) when you want backup/HA automation; pick Fly only if you accept option-A-grade DB ops.

#### C. Kubernetes Postgres operator (CloudNativePG / Zalando)

| PROS | CONS |
|---|---|
| Genuine HA — the operator promotes a replica on primary loss in seconds; the gold standard for multi-team where agent fleets depend on uptime. | **Highest operational burden by far.** You must run + understand k8s, the operator, object-storage backup config, and PVC/storage-class semantics — wildly disproportionate for one operator. |
| Declarative + GitOps-able cluster spec (primary, replicas, backup schedule, PITR retention) — strongest fit for the eventual multi-company instance. | The repo is **not k8s-shaped today**: one `Dockerfile`, docker-compose files, **zero manifests/Helm charts.** Adopting C means net-new infra authoring before any value lands. |
| Built-in continuous backup to object storage (Barman/CloudNativePG) → real PITR without hand-scripting `pg_basebackup` + WAL archiving. | The boot-time auto-migration race (`index.ts:135`, no advisory lock) becomes a **LIVE hazard** the moment the Deployment scales past `replicas:1` — you MUST gate migrations + run a one-shot Job + add a blocking `pg_advisory_lock` on the dedicated `max:1` connection (synthesis §8.2) before scaling pods. |
| Read replicas are first-class (read-only Service) — the exact substrate option D needs. | CloudNativePG's pgbouncer in transaction mode re-triggers the prepared-statement problem (`client.ts` 6543-only) AND the `SET LOCAL`-in-txn RLS requirement (`summarizer-queue.ts:15-19`). Both must be solved before this topology is safe pooled. |
| Connection pooling integrates cleanly via the managed pgbouncer `Pooler` CRD — the correct self-hosted Supavisor replacement under the no-third-party constraint. | Cost/complexity (control plane + nodes + storage) is the highest of all options; the failure surface (operator bugs, PVC issues, etcd) is larger. |

**Best when:** **only** at true multi-team scale, when you already operate Kubernetes, and HA + RLS + per-tenant isolation are hard requirements together. **Explicitly deferred** — not a starting topology. Adopt as the Phase-4 destination iff the cluster operational cost is already being paid for other reasons.

#### D. Primary + read-replica (overlay on B or C)

| PROS | CONS |
|---|---|
| Targets the real bottleneck: retrieval runs on **every agent heartbeat** for every active agent (`loadCandidates`/`queryRanked`, `memory.ts`, fired ~`heartbeat.ts:3895-3947`), so a replica absorbs read load without touching the write path. | Requires an **app-level read/write split that does not exist today**: `createDb` (`client.ts:50`) returns a single connection; there is no replica-endpoint notion. Net-new wiring, and a careless split sends a write to the replica and fails. |
| Keeps the primary's connection budget + CPU for writes/migrations/governance. | **Replica lag** makes a just-written human-answer/PR-approval briefly invisible to retrieval — for a hallucination-minimizing store this is a correctness wrinkle. A **read-your-writes** rule must route freshly-captured entries to the primary (or accept a small staleness window). |
| pgvector ANN (HNSW) scans are read-only + CPU-heavy → exactly the workload to push to a replica once `0052` lands. | Doubles (or more) DB cost + the connection budget; every replica multiplies `app_pool_max × replicas` in the budget math. |
| Incremental + reversible — add the replica only when read metrics demand; route reads back to the primary with a config change. | The future RLS phase must apply identically on the replica (policies replicate, but the `BYPASSRLS` role + `SET LOCAL`-in-txn must be validated on the read path too). |
| On RDS/Cloud SQL a replica is a console action; on CloudNativePG a replica-count bump — low marginal effort on an existing primary. | **Premature here:** at one operator + small team the primary alone is wildly over-provisioned for the read load. Adding a replica now is cost with no benefit. |

**Best when:** late team phase, gated on a **measured trigger** (retrieval read latency or primary CPU sustained high). Layer on top of B or C; never the starting point. Pair with the read/write split wiring and a read-your-writes rule for freshly-captured human-sourced memory.

## 2. Ops & connections — option families

The topology above sits on four cross-cutting operational axes. Each has a *today* choice and a *team* delta. None of the team deltas forces a rewrite, because every bridge choice (advisory lock on the dedicated connection, `prepare:false` detection, app-layer `companyId`) is forward-compatible with the team choice.

### 2.1 Migrations

| Option | PROS | CONS | Best when |
|---|---|---|---|
| **App-auto-apply-on-boot** (today's default — `index.ts:135` → `applyPendingMigrations`, `client.ts:647`, on a `max:1` connection, **no advisory lock**) | Zero operational steps; fresh clones self-heal schema (`index.ts:129-132`); has drift-repair (`reconcilePendingMigrationHistory`) + idempotent fallback; runs on its own `max:1` connection so it doesn't borrow the app pool. | **DANGEROUS for multi-replica:** no cross-process lock → N replicas booting together race on the same DDL (trust-spine `0049`, owner_id `0048`). Silent headlessly (`index.ts:135`) — no gate. Empty-central-DB boot silently migrates to an empty schema if the ETL hasn't run. | Single-operator, single-replica, embedded or one external Postgres. Set `COMBYNE_MIGRATION_AUTO_APPLY=true` to make the headless behavior **explicit, not incidental** (the flag already exists, `index.ts:134`). |
| **In-boot, wrapped in a blocking `pg_advisory_lock`** (the bridge) | Lets you stay single-binary today and add replicas later with **no re-architecture**. Reuses the existing `max:1` dedicated-connection pattern (`client.ts:651`) where the lock is session-stable. No CI infra for small teams. Blocking (not `try`) → racing replicas wait rather than skip. | **Sharp correctness edge:** if a refactor moves the lock onto a pooled/shared connection it silently stops serializing — the exact `summarizer-queue.ts:17` backend-pinning trap — giving FALSE confidence. The lock MUST be on the migration's own reserved `max:1` connection, and a test must prove two concurrent boots serialize. Still couples boot to DDL duration; serializes but does not *review* a bad migration. | The transition zone: a single operator who will grow to 2-3 replicas before standing up CI. Ship as the forward-compatible default. |
| **One-shot CI/job `pnpm db:migrate`** (`packages/db/src/migrate.ts`, `db:migrate` script already wired) against the direct `:5432` | Eliminates the multi-replica boot race entirely — exactly one job applies DDL; replicas only read a migrated schema. Decouples migration failure from app availability. DDL not at the mercy of a transaction pooler (runs against direct `:5432`). Forces conscious schema promotion (STRICT posture). | Net-new operational wiring (CI step / init-container + ordering). Requires a **new `COMBYNE_RUN_MIGRATIONS_ON_BOOT` gate** (does not exist today) so the app doesn't also migrate. Two endpoints to manage. If the job is skipped, the app boots un-migrated and fails at query time. | Multi-replica team scale, or any time 2+ app instances can boot concurrently. The team delta on top of the bridge. |

### 2.2 Pooling

| Option | PROS | CONS | Best when |
|---|---|---|---|
| **postgres-js built-in pool** (today — `createDb`, `client.ts:50-53`, **unsized** → silent `max:10`; `prepare:false` only on `port===6543`) | Zero extra infra. `prepare:true` retained for direct connections. Works fine against a direct self-hosted `:5432` for one app process. | Silent `max:10` is invisible until replicas multiply `10×replicas` + the ~6 maintenance `max:1` paths and starve your own `max_connections`. Pooler detection is Supabase-specific — a self-hosted pgbouncer on `:6432` or host `pgbouncer` gets `prepare:true` and breaks under transaction pooling. No `connect_timeout` → a dead DB hangs boot. `doc/DATABASE.md:89-96` hand-instructs editing `createDb` to unconditional `{prepare:false}`, which **disagrees with the code** — a documented footgun. | Single operator, single app process, direct Postgres. Set explicit `COMBYNE_DB_POOL_MAX`/`idle_timeout`/`connect_timeout` so defaults aren't a surprise at scale. |
| **External pgbouncer — TRANSACTION mode** | Highest connection density (many replicas behind few backends); you own `max_connections` instead of a SaaS cap; pgbouncer absorbs the fan-out — the self-hosted Supavisor replacement under the no-third-party constraint. Keeps RLS scalable (small backend count bounds per-session GUC overhead). | **BREAKS RLS via plain `SET`** — a `SET app.current_company` on checkout may not share a backend with the subsequent query (same hazard as `summarizer-queue.ts:17`) → RLS scoping requires `SET LOCAL` inside an explicit per-request txn, or a session-mode endpoint. Breaks session-scoped advisory locks + session GUCs generally. Loses prepared-statement plan caching (`prepare:false`). New component to run/monitor/rotate (`userlist.txt`). | Multi-team/multi-replica scale where connection count against your own Postgres is the constraint and RLS is enabled. Pair with a SEPARATE session-mode endpoint for RLS-scoped + `BYPASSRLS` traffic. |
| **External pgbouncer — SESSION mode** | RLS-safe with plain session GUCs; session advisory locks + prepared statements behave like a direct connection. Drop-in: `prepare` can stay `true`; the per-request RLS `SET LOCAL` and the `BYPASSRLS` scheduler both work. Simplest correct option for RLS/background traffic. | Far lower multiplexing — a backend held per session barely raises the ceiling over direct connections. Not the right primary for high-fan-out app traffic. Still a component to operate. | As the SECONDARY endpoint alongside transaction-mode: route RLS-scoped per-request traffic + the `BYPASSRLS` heartbeat scans here; keep bulk app traffic on transaction-mode. |

**Budget math (replaces the Supabase 200-conn cap — WE own this):**
`app_pool_max × replicas + maintenance(max:1 × ~6 paths: inspectMigrations, applyPendingMigrationsManually, reconcile, migratePostgresIfEmpty, ensurePostgresDatabase, plus the migration job) + headroom < our chosen max_connections`. Let pgbouncer absorb the multiplexing so Postgres backends stay low. Nothing in the code computes or enforces this automatically — it must be computed and enforced per-deploy.

### 2.3 Backups / PITR

| Option | PROS | CONS | Best when |
|---|---|---|---|
| **`pg_dump` on a cron** (today — `db:backup`, config-driven via `databaseBackupEnabled`/`IntervalMinutes`/`RetentionDays`, `config.ts:49-51`) | Already implemented + config-driven; zero net-new for one operator. Portable logical format — doubles as the input to the embedded→central memory ETL cutover. Simple to verify (restore into a scratch DB). | **No PITR** — you lose everything between dumps. For STRICT human-curated knowledge (the highest-value, least-regenerable data in the system) that is real data loss. Locks/contention as the DB grows. Today's script is embedded-port-oriented (`embeddedPostgresPort` default 54329) — needs to target the external `DATABASE_URL` host for the central DB. | Single-operator central DB, and as the portable logical export everywhere (keep it even at team scale as the ETL/portability artifact). |
| **WAL archiving + base backups (PITR)** (pgBackRest or `pg_basebackup` + `archive_command`) | True PITR — recover to any moment; the right protection for human-gated knowledge you cannot regenerate. Bounded data loss (seconds, not a full cron interval). Standard self-hosted tooling; you own retention + storage. | Real operational weight (archive storage, monitoring archive lag, restore drills) — overkill for one operator. Restore is more involved than reloading a dump. Needs storage you provision + pay for. | Team scale / once the curated knowledge base is load-bearing. The backups delta; keep `pg_dump` as the logical companion. |
| **Managed snapshots (RDS / Cloud SQL / operator-PITR)** | PITR with near-zero operational code — the platform handles WAL/snapshots/retention. Honors the locked decision (user owns the account; no third party holds data). Frees the team from running pgBackRest. | Couples you to the platform's snapshot model + restore UX. Cross-account/portability still needs the `pg_dump` logical export. Cost scales with retention window. | Team scale when the central Postgres already lives in a cloud account the user controls and you'd rather not run pgBackRest by hand. |

### 2.4 Secrets / connection-string

| Option | PROS | CONS | Best when |
|---|---|---|---|
| **env / file-config `DATABASE_URL`** (today — `databaseUrl` from env or `fileDbUrl`, `config.ts:241`; `docker-compose.yml` inlines `DATABASE_URL` + `BETTER_AUTH_SECRET` as plaintext env) | Trivial for one operator. The `local_encrypted` master key (`DATABASE.md:133`) already protects `company_secrets` at rest. `COMBYNE_SECRETS_STRICT_MODE=true` can block new inline sensitive env. | The DB password + `BETTER_AUTH_SECRET` sit in **plaintext** compose env/`config.json` — not in any secret store; fine for one box, not for a multi-team instance. No role separation: one connection string for app + maintenance + migration → no least-privilege, no `BYPASSRLS`-vs-app split. Rotation = editing files + restarting. | Single operator on a box they control. Acceptable now; explicitly the thing to replace before onboarding a team. |
| **External secret store + least-privilege roles** (Docker/K8s secrets, SOPS, or cloud secret manager; distinct RLS-enforced app role / `BYPASSRLS` scheduler role / migration-only role; rotate via pgbouncer `userlist`) | No plaintext credentials in compose/config — required for a shared instance. Role separation enforces the STRICT/RLS model (app role cannot bypass RLS; only the named scheduler role can). Rotation without app restarts via pgbouncer `userlist`. | Net-new wiring (secret-store integration + role provisioning). More moving parts to audit. Overkill for one operator. | Multi-team onboarding — the secrets delta that ships **alongside** RLS + pgbouncer. Provision the `BYPASSRLS` scheduler role **in the same change** as RLS, or the heartbeat global scan (`heartbeat.ts:6559`) silently returns zero rows. |

## 3. Isolation & vector — option families

### 3.1 Isolation (tenancy)

Tenancy is locked **multi-team**, so RLS is mandatory — the only question is *when* it lands. Three positions:

| Option | PROS | CONS | Best when |
|---|---|---|---|
| **A1 — App-layer `assertCompanyAccess` only** (status quo, zero RLS — `authz.ts:18-31`; memory reads hard-filtered by `eq(memoryEntries.companyId, companyId)` inside `loadCandidates`/`listEntries`, `memory.ts:303`, `:326`) | Zero new infra; works today in single-operator `local_trusted` mode where there is exactly one tenant. **The three self-hosted RLS interaction risks (named blocking prerequisites, not discoveries):**

1. **`BYPASSRLS` role.** The heartbeat global scan `db.select().from(agents)` (`heartbeat.ts:6559`) has NO company filter; the new memory ETL + `runDecayPass` run outside any request actor. Under RLS these silently return **zero rows** and instance-wide processing halts unless they run as a WE-created `CREATE ROLE … BYPASSRLS` scheduler role (and the APP role does NOT have `BYPASSRLS`). The role must be provisioned **in the same change** as the RLS migration.
2. **`SET` vs `SET LOCAL` on the pooler.** There is **no per-request transaction today** (`actorMiddleware` sets `req.actor` then `next()` on the shared pool, `client.ts:50`). A plain `SET app.current_company` on checkout is unsafe under pgbouncer **transaction** mode — `SET` and query may land on different backends (the `summarizer-queue.ts:15-19` pinning hazard). Fix: `SET LOCAL` inside an explicit `db.transaction()` that also contains the query, and on self-hosted route RLS-scoped traffic through pgbouncer **SESSION** mode (or direct `:5432`), keeping transaction mode only for stateless traffic. WE own the `max_connections` + pool math — no Supabase cap, no Supavisor.
3. **`local-board`-owned rows become unreachable** under company-keyed RLS unless the ETL **owner-remap** (`local-board → <userId>`, synthesis §7/§8.4) ran first. Sequence the remap before enabling policies.

**Paired workstream:** the agent JWT is signed by a **single global secret** (`agent-auth-jwt.ts:97-109`) and the middleware trusts `company_id` verbatim. A leaked secret forges any tenant's claim, and **RLS does NOT rescue you** because the app sets the tenant GUC from that same trusted claim. Per-tenant key separation must land **with** RLS, not be assumed covered by it.

## 4. Hard criticism of the recommended stack (its failure risks)

The recommendation below is **A → B for topology, A3 for isolation, B3 for vector.** Here is where it can fail. These are not hypotheticals — each is a flagged failure mode in the synthesis or a verified code drift in this repo.

1. **The A→B recommendation leans on a clean cutover ETL that does not exist yet.** `company-portability.ts` carries **no** memory tables (config-only bundle, synthesis §8.4), so switching `DATABASE_URL` today silently boots an **EMPTY** central DB and loses all dogfooded memory. The `db:memory-export`/`db:memory-import` scripts (synthesis §8.4) are a **hard prerequisite for BOTH A and B** — if they slip, the topology recommendation produces **silent data loss, not a migration.** *Mitigation that must be enforced:* the cutover doc **refuses to proceed without a verified non-empty import** (count assertion against the source DB).

2. **Recommending A first assumes single-replica forever during the A phase.** The moment the operator runs even two app containers against the same VM Postgres (e.g. for a zero-downtime restart), the boot-time auto-migration race (`index.ts:135`, no advisory lock) becomes **live** — the same hazard I'm crediting A for avoiding. *Therefore the `COMBYNE_RUN_MIGRATIONS_ON_BOOT` gate + the advisory-locked one-shot migrate Job should arguably land in the A phase as a guardrail*, not be deferred to B, or someone WILL scale to 2 replicas without it.

3. **A's backup story is the weakest link and the recommendation can state it but not enforce it.** "WE own backups/PITR" on a single VM is only as good as an actually-tested WAL-archive + restore drill — and most single-VM setups never test restore. STRICT human-gated knowledge is the highest-value, least-regenerable data in the system from day one. *Honest framing:* A is fine for a **SHORT** window; do not park a team's knowledge base on un-restore-tested A for months. There is a real argument that WAL/PITR should NOT wait for team scale precisely because the curated knowledge is irreplaceable.

4. **Choosing B quietly assumes the user is on AWS or GCP.** On a provider without an RDS/Cloud SQL equivalent (or ideologically on-prem), B collapses toward A-with-extra-cost or forces C earlier than ideal. *"Which cloud are you on?" is a gating question*, not a presumed hyperscaler.

5. **"Managed in your own account" spans a durability spectrum.** From RDS Multi-AZ (excellent) to **Fly Postgres (you own backups)**. Lumping them as one option B risks the user picking the cheap end and inheriting A's liabilities while believing they bought B's safety. *Pick RDS or Cloud SQL for the backup/HA automation; pick Fly only if you accept option-A-grade DB ops.*

6. **The whole A→B→(C/D) path assumes the pooler/prepared-statement and RLS-`SET LOCAL` prerequisites get done BEFORE pooling/multi-tenancy, not after.** History shows these cross-cutting fixes get skipped: `doc/DATABASE.md:89-96` already hand-instructs editing `createDb` to unconditional `prepare:false` while `client.ts:14-17` only does it for `port===6543` — **doc and code already disagree.** If skipped: B-with-pgbouncer silently breaks prepared statements, and the future RLS phase silently returns **zero rows** under transaction pooling — both are the synthesis's own flagged failure modes.

7. **A3's correctness depends entirely on the trigger firing.** If team onboarding is rushed and the RLS gate is skipped, you ship multi-tenant on app-layer scoping that is **one forgotten `WHERE` clause** from a cross-tenant leak. The gate MUST be a CI/cutover **hard-stop** (refuse-to-proceed without RLS + a completed `BYPASSRLS` audit), not a checklist line.

8. **The `BYPASSRLS` audit is unbounded by nature.** `heartbeat.ts:6559` is the obvious global scan, but the summarizer, `runDecayPass`, the new memory ETL, health checks, and any future cross-company analytics each independently break to **zero-rows** under RLS. Missing even one means **silent partial processing in production** — no error, just empty results, the most dangerous failure class.

9. **B3's privacy claim rests on actually running a self-hosted embedder** — which adds a GPU/CPU inference service we must keep available, batch, and fall back from. If the embedder is down and writes block on embedding, the **capture hooks fail**; if writes proceed without embedding, the entry is **unretrievable until backfilled.** The async/fallback design is load-bearing and easy to under-build.

10. **Owning `max_connections` removes the SaaS guardrail but not the failure mode.** Without pgbouncer, `COMBYNE_DB_POOL_MAX × replicas + ~6 maintenance max:1 paths + the migration job` can still exhaust a modest self-hosted Postgres (default `max_connections=100`), and the symptom is opaque `connection-refused` under load. The budget math is only protective if someone actually computes + enforces it per-deploy — **nothing in the code does this automatically.**

## 5. Recommended stack + decision

**Topology:** **START with A** (Docker-Compose Postgres on a single VM), then **GROW into B** (managed Postgres in the user's own AWS/GCP account — RDS or Cloud SQL, **not** Fly Postgres). Keep **C** (k8s operator HA) and **D** (read-replica) as explicitly-deferred, trigger-gated upgrades.

**Isolation:** **A3** — RLS at the team-onboarding boundary, authored + CI-tested on the single tenant during the central-DB phase, flipped at the first multi-tenant event as a refuse-to-proceed gate. Until then, harden the one real app-layer gap (fail-closed on empty/undefined `companyId` for admin/`local_implicit`, `authz.ts:25-30`).

**Vector:** **B3** — pgvector + a **self-hosted** real embedder (default 1536), with the irreversible HNSW-dimension choice deferred via a NULLABLE `vector(N)` column and the index built **last**, behind `COMBYNE_VECTOR_SEARCH_ENABLED`. Keep the FNV-1a hash-64 jsonb path as the deterministic test oracle.

**Ops baseline NOW vs team DELTA:**

| Axis | NOW (single-replica A) | TEAM DELTA (B/C) |
|---|---|---|
| Migrations | Keep app-auto-apply, made **explicit** (`COMBYNE_MIGRATION_AUTO_APPLY=true`). Land the advisory-lock-on-the-reserved-connection bridge as a guardrail (criticism #2). | One-shot `pnpm db:migrate` Job against direct `:5432` + `COMBYNE_RUN_MIGRATIONS_ON_BOOT=false`. |
